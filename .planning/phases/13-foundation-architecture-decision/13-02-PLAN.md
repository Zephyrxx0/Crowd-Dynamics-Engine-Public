    Create `src/stores/liveStore.ts`:

    ```typescript
    import { createStore } from "zustand/vanilla"
    import { useStore } from "zustand"

    import { createSimSlice, type SimSlice } from "./slices/simSlice"
    import { createMatchSlice, type MatchSlice } from "./slices/matchSlice"
    import { createWeatherSlice, type WeatherSlice } from "./slices/weatherSlice"
    import { createAlertSlice, type AlertSlice } from "./slices/alertSlice"
    import { createChatSlice, type ChatSlice } from "./slices/chatSlice"

    export type LiveStore = SimSlice & MatchSlice & WeatherSlice & AlertSlice & ChatSlice

    export const liveStore = createStore<LiveStore>()((...a) => ({
      ...createSimSlice(...a),
      ...createMatchSlice(...a),
      ...createWeatherSlice(...a),
      ...createAlertSlice(...a),
      ...createChatSlice(...a),
    }))

    export function useLiveStore<T>(selector: (state: LiveStore) => T): T {
      return useStore(liveStore, selector)
    }
    ```

    **Key patterns (per RESEARCH.md):**
    - Uses `createStore()` from `zustand/vanilla` (not `create()` from `zustand`) — vanilla store for non-hook access.
    - The `LiveStore` intersection type combines all 5 slice interfaces.
    - Spread operator `...a` passes `(set, get, api)` tuple to each `StateCreator`.
    - `useLiveStore` is a thin wrapper using `useStore(liveStore, selector)` — the React bridge.
    - The store directory sits at `src/stores/liveStore.ts` and `src/stores/slices/` — alongside existing `src/hooks/` stores (unchanged v1).
    
    Per agent's discretion (CONTEXT.md): Devtools middleware is intentionally omitted from the initial store. Future phases can wrap `createStore()` with devtools conditionally if debugging needs arise. No `persist` middleware per D-07.

<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Zustand store ↔ React components | Store state accessed via selectors; stale/wrong selectors cause rendering errors |
| Cross-slice reads via get() | Slices reading other slices' state without explicit type safety |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-04 | Tampering | Store slice state | mitigate | Each slice defines explicit initial state with typed interfaces. No `any` types used. |
| T-13-05 | Information Disclosure | Cross-slice state access | accept | The `StateCreator<LiveStore, ...>` generic intentionally enables cross-slice reads via `get()`. This is by design (chat needs match state). Access is controlled via typed selectors. |
| T-13-06 | Denial of Service | Unlimited alert/chat arrays | accept | Phase 13 stores define append semantics; downstream phases will implement caps (e.g., alert max 100, chat max 10) when they wire real data. Array growth within a session is bounded by user interaction patterns. |
