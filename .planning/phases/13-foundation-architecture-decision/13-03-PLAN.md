    ```typescript

      // Phase 15: Claude analysis loop streaming SSE alerts
      const encoder = new TextEncoder()

      const stream = new ReadableStream({
        async start(controller) {
          controller.enqueue(encoder.encode("data: placeholder\n\n"))
          controller.close()
        },
      })

      return new Response(stream, {
        headers: {
          "Content-Type": "text/event-stream",
          "Cache-Control": "no-cache",
          Connection: "keep-alive",
        },
      })
    }
    ```
    Per D-09: SSE via native ReadableStream. No nginx proxy to interfere. SSE messages use `data: ...\n\n` format.

    **src/app/api/chat/route.ts** — SSE skeleton (POST)
    ```typescript
    import { type NextRequest } from "next/server"

    export async function POST(request: NextRequest) {
      // Phase 16: receive user message, stream Claude response via SSE
      const encoder = new TextEncoder()

      const stream = new ReadableStream({
        async start(controller) {
          controller.enqueue(encoder.encode("data: placeholder\n\n"))
          controller.close()
        },
      })

      return new Response(stream, {
        headers: {
          "Content-Type": "text/event-stream",
          "Cache-Control": "no-cache",
          Connection: "keep-alive",
        },
      })
    }
    ```
    POST receives the user's chat message in the body; the response streams back via SSE. The skeleton ignores the body for now.

    **Key patterns per D-08, D-09:**

    - `NextRequest` type from `next/server` on routes that will parse query/body.
    - SSE endpoints: `new Response(stream, { headers: { "Content-Type": "text/event-stream" } })`.
    - No Express, no `NextResponse` — standard `Response` objects.
    - Route handlers are server-only — no client imports (mitigates T-13-02).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP request → Route handler | Untrusted external input enters server-side code |
| Route handler → External API (future) | Server-side fetch carries API keys |
| Route handler → Client (SSE) | Server-side data delivered to browser via stream |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-02 | Information Disclosure | Route handler ↔ client boundary | mitigate | Route handlers are pure server code with no client-importable modules. Future API key handling stays entirely server-side. Repeated from Plan 01/Plan 02 for visibility since route handlers are the attack surface. |
| T-13-07 | Denial of Service | SSE ReadableStream never closing | accept | Skeleton closes stream immediately. Phase 15/16 will implement reconnection logic and Cloud Run timeout configuration. |
| T-13-08 | Spoofing | Request body on POST /api/chat | mitigate | Phase 16 will validate request body against `ChatRequestSchema` (Zod) before processing. Skeleton ignores body. |
